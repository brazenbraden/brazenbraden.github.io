Version 0.3 of *When The Mud Came* doesn’t look like a flashy update on the surface. There’s no sudden explosion of new content, no dramatic visual overhaul, no “look at all these new things” moment.

Instead, this update is about something far less visible but far more important in the long run: **removing bespoke logic and replacing it with systems that scale**.

This is the release where the game stopped being a collection of clever one-offs and started becoming a world that can actually support stories, interactions, and progression without collapsing under its own weight.

## The problem with “just make it work”

Early in a project, hard-coding is tempting. Sometimes it’s even the right call.

You want an endgame? Fine — wire it directly to “100% clean”.
You want a readable to unlock something? Add a conditional.
You want a secret notification? Fire a toaster and move on.

That approach got the demo to a playable state, and that’s not nothing. But it also created a quiet problem: every new feature wanted its *own* logic, *its own* rules, and *its own* UI glue.

By the time version 0.2 shipped, the warning signs were clear:
- Interactions were limited to “read a thing”
- Quests only existed as a hard-coded endgame flow
- Notifications were duplicated and narrowly scoped

It all worked — but none of it wanted to grow.

Version 0.3 is where I stopped adding behaviour and started fixing the *shape* of the game.

## Interaction: from reading objects to reacting worlds

Originally, interaction meant exactly one thing: press a button, read some text.

That’s fine for lore, but it’s not a living world.

In v0.3, the interaction system was rebuilt so that **interaction is no longer the outcome — it’s the trigger**. An interaction can now fan out into effects, rather than terminating in a single hard-coded response.

Right now, that includes things like:
- Toggling lights
- Playing animations (for example, spinning playground equipment)
- Triggering transitions
- Emitting particle effects
- One-shot or disposable interactions

These aren’t theoretical. They’re already in the v0.3 build, deliberately added to stress-test the system.

That’s the difference between a prop you can click and a world that responds.

## Questing: killing the bespoke endgame

Before v0.3, the game technically had a quest — but only in the loosest sense.

The endgame flow was completely bespoke:
- World reaches 100% clean
- Specific readables must be read
- Vehicle unlocks
- Game ends

It was tightly coupled, purpose-built, and impossible to reuse without copying logic.

In v0.3, that entire approach was replaced with a **generic questing system**, built from composable parts:
- Tasks
- Task definitions
- Conditions
- Sequences

Instead of encoding *what happens*, the system defines *what progress looks like*.

![Quest System]({{site.images}}/whenthemudcame/v0.3/quest.jpg)

At the moment, the system powers:
- The main endgame flow
- One side quest

That’s not because the system is limited — it’s because the demo map is small. There simply isn’t space yet for sprawling questlines. The important part is that the scaffolding now exists.

This is the moment where “the endgame” stopped being a special case and became just another quest.

That matters more than adding ten new objectives ever could.

## Notifications: one feedback layer, not three half-systems

Notifications were quietly one of the messiest parts of the game pre-v0.3.

There were multiple systems doing similar things:
- A general toaster
- A separate secret discovery notification
- Ad-hoc logic to tie them into gameplay events

In v0.3, this was unified into a **single, generic, event-driven notification system**.

Now:
- Any interaction can raise a notification
- Any quest event can surface feedback
- Notifications can be customised per use case
- Notifications themselves can trigger follow-up behaviour if needed


This wasn’t about visuals — it was about architecture.

As more systems come online, the last thing the game needs is five different ways of telling the player that something happened. A single feedback layer keeps the experience readable, predictable, and extensible.

---

## What version 0.3 is really about

Version 0.3 isn’t a content update. It’s a **capability update**.

It removes special cases.
It replaces hard-coded flows with authorable structures.
It shifts the game toward an event-driven spine where systems talk to each other cleanly.

Most importantly, it gives future-me a fighting chance.

From here on:
- Adding new interactions doesn’t mean rewriting logic
- Adding quests doesn’t mean inventing new frameworks
- Adding feedback doesn’t mean duplicating UI systems

This is the point where the game stopped asking “does this work?” and started asking “will this still work in six months?”

That’s what version 0.3 brings to the table — even if most players will never see it directly.

And that’s exactly how it should be.
